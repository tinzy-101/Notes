System models:

  • non virtual machine: hardware --> kernel --> processes
  • virtual machine: hardware --> kernel --> processes (several kernels and process threads)

History:

  • ibm started vms so systems guys could test their solutions to bugs in vms, and not have to work odd hours to fix these problems

Benefits and Features

  • host system protected from VMs, VMs protected from each other
  • freeze, suspend running VM
  • good for OS dev and research

Trap and emulate:

  • how to switch from virtual user mode to virtual kernel mode?
    • use trap-and-emulate: VMM gain control, analyze error, and execute operation, then return controll to VM in user mode

binary translation:

  • when trying to run an OS on a different host, it might have diff CPU (diff machine instruction sets)
  • need to use emulation to perform binary translation of instrucion sets

nested page tables:

  • VM needs to manage pages and page tables (2 OSes thing they own the page tables)
  • can have control in hardware
  • can have many levels of virtual addressing

types of VMs and implementations:

  1. type 0 hypervisor
    • VMM is in firmware, OS doesn't need to do anything special
  2. type 1 hypervisor
    • have series of OSes linked together and have cloning
    • run in kernel mode
    • they don't know they're running VM
    • moving towards hardware
  3. type 2 hypervisor
    • little OS involvement in virtualization
    • poor overall performance
    • don't take advantage of CPUs
  4. paravirtualization
    • VMM provides services that guest must be modified to used
    • icnreased performance
    • almost virtualization (para)
    • xen = very good example
    • emulation != virtualization!!
